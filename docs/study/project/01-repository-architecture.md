# Общая архитектура репозитория

## Что это за репозиторий по факту

`airflow-dwh-proj` — не один линейный пайплайн, а учебный репозиторий с общей инфраструктурой и двумя разными практиками внутри одного контура.

Общая идея репозитория такая:

- Airflow оркестрирует задачи;
- PostgreSQL хранит данные;
- Spark вынесен в отдельный вычислительный контур;
- Grafana поднята как отдельный слой визуализации;
- sample-данные и генератор лежат прямо в репозитории.

Важно сразу зафиксировать, что здесь не один DWH-сценарий.
Внутри репозитория сосуществуют два разных подхода:

- упрощенный контур `raw -> mart` со Spark;
- более классический контур `stg -> dds -> dm` без Spark.

## Как устроено дерево репозитория

### `airflow/`

Каталог `airflow/` — это основной инфраструктурный и orchestration-контур.
Здесь находятся:

- `docker-compose.yaml` с Airflow, Redis, PostgreSQL, отдельным `postgres-dwh` и Grafana;
- `Dockerfile` кастомного образа Airflow;
- `requirements.txt` с Airflow providers и `pyspark`;
- `init.sh` для локальной подготовки `.env` и JDBC jar;
- `dags/` с обоими DAG-ами, SQL и общими константами;
- `scripts/` с SQL-инициализацией баз.

### `spark/`

Каталог `spark/` — это отдельный вычислительный контур.
В нем лежат:

- `docker-compose.yaml` c `master` и `worker`;
- `Dockerfile` образа Spark;
- `apps/process_data.py` — Spark job для practice 1;
- `apps/postgresql-42.7.3.jar` — JDBC driver для PostgreSQL.

### `dataset/`

`dataset/` содержит sample CSV для CRM-практики.
Это не автоматически используемый рабочий слой Airflow, а репозиторный набор примеров, который можно копировать в `airflow/dags/input/`.

### `data_generator.py`

Это вспомогательный генератор данных для practice 2.
Он умеет создавать:

- мастер-данные клиентов и кампаний;
- дневные факты по рекламе;
- дневные факты мониторинга сайтов.

## Две практики внутри одного проекта

## Практика 1: `elt_pipeline`

Первая практика строится вокруг DAG-а `elt_pipeline`.
Ее логика максимально короткая:

1. создать `raw`-таблицы;
2. создать одну `mart`-таблицу;
3. сгенерировать тестовые данные в PostgreSQL;
4. вызвать Spark job;
5. записать итоговую витрину обратно в PostgreSQL.

Это учебный контур, который показывает, как Airflow вызывает Spark и как Spark работает с PostgreSQL по JDBC.

## Практика 2: `crm_report_daily`

Вторая практика устроена существенно богаче.
Она использует:

- CSV как вход;
- `stg` для приемки;
- `dds` для справочников и фактов;
- `dm` для витрин;
- отдельную базу `postgres-dwh`;
- потенциальное потребление данных из Grafana.

Эта практика ближе к учебной мини-версии классического DWH-контура.

## Самое важное архитектурное различие между практиками

### Practice 1

- использует Spark;
- работает со схемами `raw` и `mart`;
- хранит данные в сервисе `postgres`, который одновременно является metadata DB Airflow.

### Practice 2

- не использует Spark;
- работает через `pandas`, SQL и `PostgresHook`;
- использует отдельную базу `postgres-dwh`;
- строит слои `stg -> dds -> dm`;
- логически ближе к будущему использованию в Grafana.

Это ключевая особенность репозитория.
Если читать проект как один одинаковый пайплайн, очень легко запутаться.

## Общий инфраструктурный контур

Обе практики живут на общей контейнерной основе:

- Airflow cluster c `CeleryExecutor`;
- Redis как broker;
- PostgreSQL для Airflow и practice 1;
- отдельный PostgreSQL DWH для practice 2;
- Grafana;
- отдельный Spark cluster.

Связующим слоем между compose-контурами выступает сеть `shared-data-network`.

## Что в этом репозитории сделано учебно и упрощенно

Есть несколько решений, которые нужно понимать именно как учебные:

- practice 1 хранит бизнес-данные рядом с metadata DB Airflow;
- `mart.client_spend_report` пополняется в режиме `append` и не защищен от повторной записи;
- Grafana присутствует как сервис, но ее datasource и dashboards не описаны декларативно;
- в practice 2 staging-слой полностью текстовый и почти не валидирует вход;
- Spark вынесен в отдельный compose, что удобно для демонстрации границ контуров, но требует общей внешней сети и shared volume.

## Зачем репозиторий устроен именно так

С учебной точки зрения структура удачна, потому что в одном месте можно изучить сразу несколько сценариев:

- SQL bootstrap через Airflow;
- Python-функции и hooks;
- Spark submit из Airflow;
- загрузку CSV через `pandas`;
- построение `dds` и `dm`;
- работу нескольких PostgreSQL-контуров;
- роль Grafana как потенциального потребителя витрин.

Именно поэтому репозиторий лучше читать не как «один законченный production-проект», а как учебный стенд с двумя разными практиками на общей базе.
