# SQL и PostgreSQL, используемые в проекте

## Почему нужен именно такой набор SQL

В аналитическом проекте SQL используется не только для выборок.
Он одновременно решает несколько задач:

- описывает схемы и таблицы;
- контролирует ключи и ограничения;
- переносит данные между слоями;
- строит факты и витрины;
- помогает поддерживать идемпотентность повторных запусков.

Поэтому для понимания проекта мало знать `SELECT`.
Нужно разбираться и в DDL, и в DML, и в типизации, и в типовых шаблонах загрузки PostgreSQL.

## DDL: создание схем и таблиц

### `CREATE SCHEMA`

`CREATE SCHEMA` используется для логического разделения объектов.
Схема помогает держать слои данных отдельно и не смешивать, например, staging-таблицы с витринами.

Чаще всего схема решает сразу три задачи:

- группировка объектов по назначению;
- снижение риска конфликта имен;
- явная фиксация места таблицы в архитектуре.

### `CREATE TABLE`

`CREATE TABLE` задает структуру таблицы:

- колонки;
- типы данных;
- ограничения;
- значения по умолчанию.

`IF NOT EXISTS` делает DDL повторяемым.
Это важно в учебных пайплайнах, где одни и те же init-скрипты могут запускаться многократно.

## Типы данных PostgreSQL, которые реально важны

### `SERIAL`

`SERIAL` — удобный способ создать автоинкрементный целочисленный идентификатор.
В новых проектах часто предпочитают явные sequence или identity columns, но в учебном контуре `SERIAL` остается понятным и практичным вариантом.

### `VARCHAR` и `TEXT`

`VARCHAR(n)` используют там, где нужен ожидаемый предел длины строки.
`TEXT` применяют, когда строка по смыслу текстовая, а жесткое ограничение длины не критично.

В staging-слое часто используют `TEXT`, потому что вход еще не до конца типизирован.

### `DATE` и `TIMESTAMP`

`DATE` хранит только календарную дату.
`TIMESTAMP` хранит дату и время.

Различие важно, потому что витрины по дням и месяцам обычно работают с `DATE`, а технические поля загрузки и обновления — с `TIMESTAMP`.

### `DECIMAL`

`DECIMAL(p, s)` используют для точных чисел, особенно для денег и процентов.
Для аналитики это безопаснее, чем использовать float там, где нужна предсказуемая точность.

### `BOOLEAN`

`BOOLEAN` нужен для логических флагов, например активности записи или технического состояния сущности.

### `DEFAULT NOW()` и `DEFAULT CURRENT_TIMESTAMP`

Эти выражения позволяют автоматически выставлять временную метку при вставке строки.
Они особенно полезны для полей вроде `loaded_at`, `created_at`, `updated_at`.

## Ограничения целостности

### `PRIMARY KEY`

`PRIMARY KEY` однозначно идентифицирует строку.
Это основа безопасного `upsert` и понятной модели таблицы.

### `FOREIGN KEY`

`FOREIGN KEY` фиксирует связь между таблицами.
Он не дает вставить ссылку на несуществующую сущность и помогает держать модель согласованной.

### `UNIQUE`

`UNIQUE` запрещает дубли по заданному набору колонок.
Для аналитических таблиц это особенно полезно, когда уникальность строки определяется не одним surrogate key, а комбинацией периода и бизнес-ключа.

## Очистка и повторная загрузка

### `TRUNCATE`

`TRUNCATE TABLE` быстро очищает всю таблицу.
Это хороший выбор для полного сброса небольшого слоя перед новой загрузкой.

### `RESTART IDENTITY`

`RESTART IDENTITY` сбрасывает sequence, связанный с `SERIAL`.
Это удобно в учебных сценариях, где после очистки нужно получить те же самые диапазоны идентификаторов.

### `CASCADE`

`CASCADE` распространяет действие на зависимые объекты или связанные строки.
Команда мощная, но ее нужно использовать осознанно, потому что эффект может затронуть больше таблиц, чем кажется на первый взгляд.

### `DELETE`

`DELETE` удаляет строки выборочно.
Это удобнее `TRUNCATE`, когда нужно пересобрать не всю таблицу, а только один день, месяц или другой ограниченный срез.

## Загрузка и обновление данных

### `INSERT INTO ... SELECT`

Этот паттерн нужен, когда данные переливаются из одного слоя в другой с преобразованием типов, фильтрацией, join или агрегацией.
Это один из базовых инструментов ELT-подхода.

### `UPDATE`

`UPDATE` меняет существующие строки.
В аналитических пайплайнах его часто используют для служебных признаков, например флага активности.

### `INSERT ... ON CONFLICT DO UPDATE`

Это основной PostgreSQL-шаблон для `upsert`.
Он нужен, когда строка должна либо вставиться, либо обновиться по уникальному ключу.

Такой подход хорош для:

- справочников;
- дневных витрин по стабильному ключу;
- актуального состояния сущностей.

## Соединения и агрегации

### `JOIN`

`JOIN` объединяет таблицы по ключам.
Без него невозможно собрать факты и измерения в один аналитический запрос.

### `GROUP BY`

`GROUP BY` задает уровень агрегирования.
Если грейн результирующей таблицы — «день и платформа», то именно эти поля должны участвовать в группировке.

### `SUM`, `AVG`, `COUNT(DISTINCT ...)`

Это базовые агрегаты для аналитики:

- `SUM` — сумма показателей;
- `AVG` — среднее значение;
- `COUNT(DISTINCT ...)` — количество уникальных сущностей.

Именно такие агрегаты обычно лежат в основе витрин KPI.

### `CASE`

`CASE` добавляет в SQL условную логику.
Это удобно для статусов, категорий и простых правил интерпретации данных.

## Работа с датами, типами и пустыми значениями

### `TO_DATE`

`TO_DATE` преобразует строку в `DATE` по заданному формату.
Это особенно полезно, когда сырой слой хранит дату в текстовом виде.

### `DATE_TRUNC`

`DATE_TRUNC` позволяет обрезать дату до месяца, дня и другой нужной гранулярности.
В аналитике это один из базовых инструментов для периодических витрин.

### `CAST` и `::type`

Явное приведение типов нужно, когда входные данные приходят как текст, а дальше должны участвовать в вычислениях.

Обе формы корректны:

- `CAST(value AS INTEGER)`
- `value::INT`

### `NULLIF`

`NULLIF(a, b)` превращает значение в `NULL`, если `a = b`.
Самый частый аналитический сценарий — защита от деления на ноль.

Без `NULLIF` расчет вроде `cost / clicks` может завершиться ошибкой, если `clicks = 0`.

## Почему это важно именно для аналитических пайплайнов

Аналитический SQL должен быть не просто рабочим, а воспроизводимым.
Для этого особенно важны:

- явная типизация;
- ограничения на ключи;
- безопасные агрегаты;
- понятная работа с датами;
- повторяемые паттерны `truncate`, `delete+insert` и `upsert`.

Именно этот набор конструкций обычно делает из обычной базы учебное аналитическое хранилище.
