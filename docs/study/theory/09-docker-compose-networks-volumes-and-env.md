# Docker Compose, сети, volume и локальная инфраструктура

## Зачем нужен Docker Compose

`Docker Compose` позволяет описать локальный многоконтейнерный контур в YAML-файле и поднимать его воспроизводимо.
Это особенно полезно для учебного проекта, где одновременно нужны:

- оркестратор;
- брокер сообщений;
- одна или несколько баз;
- вычислительный движок;
- слой визуализации.

Без Compose такой контур пришлось бы собирать вручную, и воспроизводимость быстро бы потерялась.

## Что описывает compose-файл

Обычно compose-файл задает:

- список сервисов;
- образ или `Dockerfile`;
- команду запуска;
- порты;
- volumes;
- сети;
- переменные окружения;
- зависимости старта и healthcheck.

## Сервисы

`Service` — логическая единица инфраструктуры, которая запускается как контейнер или группа однотипных контейнеров.
У сервиса есть своя роль и свой lifecycle.

Для аналитического учебного контура сервисами обычно являются:

- база данных;
- очередь сообщений;
- Airflow-компоненты;
- Spark master и worker;
- BI-инструмент.

## Сети

Docker-сеть нужна, чтобы контейнеры видели друг друга по именам сервисов.
Это позволяет одному контейнеру обращаться к другому не через `localhost`, а по внутреннему DNS-имени.

### Общая сеть

Если части системы разделены на несколько compose-файлов, им нужна общая сеть.
Именно она позволяет оркестратору в одном compose обращаться к Spark-сервисам из другого compose.

### External Network

`external: true` означает, что сеть должна существовать заранее и не создается автоматически текущим compose-проектом.
Так делают, когда несколько независимых compose-стеков должны оказаться в одном сетевом контуре.

## Volume

`Volume` нужен для двух главных задач:

- сохранять данные между пересозданиями контейнера;
- делиться файлами между контейнером и хостовой машиной.

### Named Volume

Named volume обычно используют для хранения данных PostgreSQL и другого состояния, которое не должно пропасть при пересоздании контейнера.

### Bind Mount

Bind mount пробрасывает в контейнер локальный каталог хоста.
Это удобно для:

- DAG-ов;
- логов;
- конфигов;
- Spark-приложений;
- входных CSV.

## `depends_on` и `healthcheck`

### `depends_on`

`depends_on` задает зависимость старта между сервисами.
Но важно помнить, что «контейнер запущен» и «сервис готов принимать запросы» — это не одно и то же.

### `healthcheck`

`healthcheck` позволяет проверять фактическую готовность сервиса.
Для баз это часто `pg_isready`, для HTTP-сервисов — запрос на endpoint здоровья, для очередей и фоновых процессов — специальные команды проверки.

Связка `depends_on` + `healthcheck` уменьшает количество ложных стартов, когда один сервис уже поднялся формально, но еще не готов к работе.

## Переменные окружения

`Environment variables` позволяют передавать конфигурацию в контейнер без изменения кода.
В аналитическом контуре через них обычно задают:

- параметры подключения к БД;
- настройки Airflow;
- логины и пароли;
- connection URI;
- системные флаги поведения сервисов.

Это удобно, потому что одна и та же кодовая база может запускаться в немного разных окружениях без переписывания файлов.

## Межсервисное взаимодействие

В Compose-среде контейнеры общаются через:

- имя сервиса;
- внутренний порт контейнера;
- общую Docker-сеть.

Именно поэтому важно понимать, что адрес внутри контейнеров и адрес для пользователя на хосте — это разные уровни.

## Что важно запомнить

- Compose делает локальную инфраструктуру воспроизводимой.
- Общая сеть позволяет связывать несколько compose-контуров.
- Named volume хранят состояние, bind mount делят файлы.
- `healthcheck` важен не меньше, чем `depends_on`.
- Переменные окружения — основной способ прокидывать connection и настройки в контейнеры.
