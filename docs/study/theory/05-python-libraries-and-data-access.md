# Python-библиотеки и доступ к данным в DAG-ах и скриптах

## Почему Python в таких проектах так важен

В аналитическом контуре Python обычно играет роль связующего слоя.
Он нужен не для того, чтобы заменить SQL и Spark, а чтобы:

- описать DAG;
- вызвать нужные операторы;
- прочитать входной файл;
- управлять соединениями;
- передать данные в БД или Spark.

В этом смысле Python в оркестрации — это язык интеграции.

## `DAG` и Python API Airflow

Airflow описывает workflow на Python.
Это означает, что DAG — не статическая YAML-конфигурация, а полноценный Python-модуль.

Через Airflow API обычно используют:

- `DAG`;
- `PythonOperator`;
- `SQLExecuteQueryOperator`;
- `SparkSubmitOperator`;
- hooks для внешних систем.

Такой подход дает гибкость, но требует аккуратности: DAG-файл должен оставаться читаемым и не превращаться в огромный монолит.

## `PythonOperator`

`PythonOperator` позволяет сделать task из обычной Python-функции.
Это удобно, когда шаг включает:

- файловую работу;
- работу с DataFrame;
- несколько вызовов SQL;
- нестандартную логику, которую неудобно выразить одним готовым оператором.

Главное ограничение — не превращать PythonOperator в место, где бесконтрольно смешаны и orchestration, и бизнес-логика, и инфраструктурный код.

## `SQLExecuteQueryOperator`

Этот оператор нужен, когда основная работа шага — выполнить SQL.
Он особенно удобен для:

- DDL-скриптов;
- небольших DML-операций;
- SQL-файлов, которые удобно хранить отдельно от кода DAG-а.

Это хороший способ держать SQL отдельно от Python там, где шаг естественно выражается запросом.

## `PostgresHook`

`PostgresHook` — один из ключевых инструментов для работы с PostgreSQL из Airflow.
Он позволяет:

- получить raw connection;
- выполнить SQL;
- получить `SQLAlchemy engine`;
- использовать конфигурацию connection из Airflow, а не зашивать ее в код.

Это удобно, потому что один и тот же hook можно использовать на разных уровнях:

- для простого `run`;
- для ручной работы через cursor;
- для интеграции с pandas.

## `SQLAlchemy engine` через Airflow hook

Когда `PostgresHook` возвращает `get_sqlalchemy_engine()`, Python-код получает удобный объект для табличной работы.
Это особенно полезно для `pandas`.

Плюсы такого подхода:

- connection живет в Airflow, а не в коде;
- можно использовать стандартный API DataFrame;
- не нужно вручную писать низкоуровневый код подключения.

## `pandas`

`pandas` удобен для небольших табличных данных:

- CSV;
- справочников;
- промежуточных подготовительных наборов;
- локальных учебных загрузок.

Типовой паттерн:

- `read_csv`;
- приведение типов или строк;
- `to_sql` в PostgreSQL.

Важно понимать ограничение: pandas работает в памяти одного процесса.
Он хорош для небольших учебных объемов, но не заменяет распределенную обработку.

## `psycopg2` и `execute_values`

`psycopg2` — низкоуровневый PostgreSQL-драйвер для Python.
Он полезен, когда нужна явная работа с транзакциями, cursor и массовой вставкой.

`psycopg2.extras.execute_values` особенно удобен для batch insert.
Вместо сотен отдельных `INSERT` можно передать список значений одной операцией.

Это дает:

- меньше round-trip к БД;
- более быстрые bulk insert;
- явный контроль над commit/rollback.

Такой подход хорошо подходит для генерации данных и массовой загрузки небольших партий.

## `SparkSubmitOperator`

`SparkSubmitOperator` нужен, когда обработка должна выполняться не в Airflow worker, а в Spark.
Airflow в этом случае не считает данные сам, а лишь отправляет Spark-приложение на выполнение.

Это важный архитектурный момент:

- Airflow orchestrates;
- Spark computes.

## `SparkSubmitHook`

`SparkSubmitHook` работает под капотом и отвечает за техническую часть запуска Spark.
Он собирает параметры подключения, конфигурацию и итоговую команду `spark-submit`.

Понимание hook особенно полезно, когда в проекте есть нестандартная настройка подключения к Spark master.

## `PySpark`

`PySpark` — Python API к Apache Spark.
Он нужен, когда данных больше, чем удобно обрабатывать в одном процессе, или когда хочется использовать распределенные DataFrame-операции.

Типовые задачи PySpark:

- чтение из JDBC;
- `join`;
- `groupBy`;
- `agg`;
- добавление колонок;
- запись обратно в БД или файловый слой.

## Как выбирать между инструментами

### Когда достаточно SQL-оператора

Если шаг естественно выражается SQL-скриптом, лучше использовать SQL-оператор, а не писать лишний Python-код.

### Когда удобен `PythonOperator`

Если нужно собрать несколько действий в одном месте, прочитать файл или применить табличную логику, PythonOperator становится естественным выбором.

### Когда нужен `pandas`

Если данные небольшие, файловые и локальные, `pandas` обычно проще Spark.

### Когда нужен `psycopg2`

Если важны ручной контроль транзакции и быстрая массовая вставка, удобнее использовать драйвер напрямую.

### Когда нужен Spark

Если шаг должен быть вычислительно отдельным, распределенным или просто архитектурно вынесенным из Airflow worker, лучше запускать Spark job отдельно.

## Что важно запомнить

- Python в таких проектах — это язык orchestration и интеграции.
- `PostgresHook` удобен и для SQL, и для получения engine.
- `pandas` хорош для небольших учебных CSV.
- `psycopg2.execute_values` полезен для batch insert.
- `SparkSubmitOperator` и `PySpark` нужны, когда вычисления вынесены в Spark-контур.
