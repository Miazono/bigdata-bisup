# Служебная спецификация оформления для сборки `.docx`

## Назначение

Этот документ задает практическую спецификацию оформления учебных материалов из `docs/study` при программной сборке в `.docx` через `python-docx`.
Цель спецификации — получить аккуратный рабочий документ с оглавлением, понятной иерархией заголовков, плотным, но читаемым набором текста и едиными правилами для списков, кода, таблиц и акцентов.

## Область применения

Спецификация рассчитана на материалы из:

- `00-glossary.md`
- `01-learning-map.md`
- `theory/`
- `project/`
- `script-guides/`

Она подходит для генерации одного общего документа или серии документов на основе одного набора стилей.

## Базовые принципы

- Верхние разделы документа начинаются с новой страницы.
- Оглавление строится автоматически, а не вручную.
- Заголовки всегда черные, жирные, без курсива.
- Основной текст должен быть компактным, но не «прижатым».
- Цветовая схема должна быть строгой: никаких голубых и болотных оттенков.
- Для терминов и code-like сущностей допустимы только 1-2 акцентных цвета, без визуального шума.
- Код, пути, имена файлов, SQL-идентификаторы и похожие сущности отображаются моноширинным шрифтом.
- Приложение можно верстать плотнее, чем основной корпус.
- Все отступы, интервалы и акценты задаются стилями и параметрами абзацев, а не пустыми строками.

## Структура документа

Рекомендуемый порядок разделов:

1. Титульный элемент или краткая шапка документа.
2. Оглавление.
3. Основные разделы по H1.
4. Подразделы по H2 и H3.
5. Приложение или приложения.

Если документ собирается из нескольких частей, каждый новый верхний раздел должен начинаться с новой страницы, кроме оглавления и, при необходимости, титульной страницы.

## Стили документа

Ниже приведен минимальный набор стилей, который стоит создать в `python-docx` и использовать последовательно.

| Стиль | Назначение | Основные параметры |
|---|---|---|
| `Document Title` | Заголовок всего документа | 16-18 pt, bold, center, black |
| `TOC Heading` | Заголовок оглавления | 13 pt, bold, black, без курсива |
| `TOC 1-3` | Элементы оглавления | 11 pt, black, с лидером точек и выравниванием номера справа |
| `Normal` | Основной текст | 11 pt, justify, межстрочный интервал 1.15 |
| `Heading 1` | Верхний раздел | 13 pt, bold, black, page break before |
| `Heading 2` | Подраздел | 11.5-12 pt, bold, black |
| `Heading 3` | Внутренний подраздел | 11 pt, bold, black |
| `List Bullet` | Маркированный список | 11 pt, hanging indent, компактные интервалы |
| `List Number` | Нумерованный список | 11 pt, hanging indent, компактные интервалы |
| `Code Block` | Многострочный код | Consolas 9.5-10 pt, светлый фон, моноширинный стиль |
| `Inline Code` | Встроенный код | Consolas 10-11 pt, без курсива, без синего цвета |
| `Path` | Пути и имена файлов | Console-style monospace, как `Inline Code` |
| `Table Header` | Шапка таблицы | 10-11 pt, bold, black, светлая заливка |
| `Table Text` | Ячейки таблицы | 10-11 pt, компактно, без лишних отступов |
| `Caption` | Подписи к таблицам и рисункам | 10 pt, black, кратко и без украшательств |
| `Appendix Heading 1` | Заголовок приложения | 12 pt, bold, black, компактнее, чем основной H1 |
| `Appendix Body` | Текст приложения | 10 pt, line spacing 1.0-1.1 |

## Параметры страницы

- Формат: A4, книжная ориентация.
- Поля: 20 мм слева, 20 мм справа, 20 мм сверху, 20 мм снизу.
- Колонтитулы: если не нужны, оставить минимальными или пустыми.
- Нумерация страниц: допускается, но не обязательна для служебного документа.

Если документ будет содержать длинные таблицы или кодовые блоки, допустимо увеличить левое и правое поля до 22-25 мм только при реальной необходимости.

## Основной текст

Для `Normal` используйте:

- шрифт: `Times New Roman` или другой серифный шрифт с хорошей поддержкой кириллицы;
- размер: 11 pt;
- цвет: черный (`#111111`);
- выравнивание: по ширине;
- межстрочный интервал: 1.15;
- интервал перед абзацем: 0 pt;
- интервал после абзаца: 0 pt или 3 pt для перехода между смысловыми блоками;
- первая строка: отступ 1.25 см;
- запрет на курсив по умолчанию.

Если абзац следует сразу после заголовка, допускается небольшой интервал после заголовка вместо дополнительной пустой строки.

## Заголовки

### Общие правила

- Все заголовки должны быть черными и жирными.
- Курсива в заголовках не использовать.
- Не применять цветные подчеркивания, декоративные эффекты и чрезмерные капители.
- Для заголовков использовать `keep with next`, чтобы заголовок не оставался внизу страницы отдельно от первого абзаца.

### `Heading 1`

- Размер: 13 pt.
- Жирность: bold.
- Цвет: черный.
- Курсив: нет.
- Интервал до: 10-12 pt.
- Интервал после: 6 pt.
- Новая страница перед заголовком: да, кроме первого содержательного блока после оглавления.

`Heading 1` должен использоваться для крупных разделов документа. Именно этот уровень считается верхним разделом и запускает новую страницу.

### `Heading 2`

- Размер: 11.5-12 pt.
- Жирность: bold.
- Цвет: черный.
- Курсив: нет.
- Интервал до: 6 pt.
- Интервал после: 3-4 pt.

### `Heading 3`

- Размер: 11 pt.
- Жирность: bold.
- Цвет: черный.
- Курсив: нет.
- Интервал до: 4 pt.
- Интервал после: 2-3 pt.

Если вложенность глубже не нужна, лучше не вводить `Heading 4` и ниже.

## Оглавление

Оглавление должно быть рабочим, то есть строиться по стилям заголовков и обновляться как поле Word.

Правила:

- строить оглавление по уровням `Heading 1`, `Heading 2`, `Heading 3`;
- размещать оглавление после титульной части и до первого `Heading 1`;
- не вносить оглавление вручную текстом;
- после генерации помечать документ так, чтобы Word предлагал обновить поля при открытии;
- использовать лидеры точек и правое выравнивание номеров страниц;
- не включать в оглавление стили кодовых блоков, списков и таблиц.

Для `python-docx` это означает, что оглавление лучше вставлять как поле `TOC` с нужным диапазоном уровней, а не как статический текст.

## Page breaks

Правила переноса страниц:

- перед каждым верхним разделом `Heading 1` ставить разрыв страницы;
- перед приложением также ставить разрыв страницы;
- не имитировать разрывы серией пустых абзацев;
- не вставлять page break между заголовком и первым абзацем раздела;
- если раздел начинается с таблицы или списка, они должны идти сразу после заголовка без визуального провала.

В `python-docx` это лучше реализовать через `page_break_before` у стиля заголовка и точечные `add_page_break()` только там, где нужен явный переход.

## Списки

### Общие правила

- Списки должны быть аккуратными и компактными.
- Между пунктами не должно быть лишних пустых строк.
- Текст пункта должен быть коротким и завершенным по смыслу.
- Если пункт очень длинный, лучше разбить его на абзац и подсписок.

### Маркированные списки

- Использовать стандартный маркер, без нестандартных символов.
- Левый отступ: около 0.9 см.
- Hanging indent: около 0.4-0.5 см.
- Размер: 11 pt.
- Межстрочный интервал: 1.15.

### Нумерованные списки

- Использовать для последовательностей, шагов и правил.
- Отступы и интервал как у маркированного списка.
- Если пункт содержит подшаги, вложенность должна быть визуально очевидной, но не чрезмерной.

### Вложенные списки

- Каждый новый уровень увеличивает левый отступ примерно на 0.6 см.
- Не уходить глубже 2-3 уровней без веской причины.
- Вложенные списки не должны ломать читаемость страницы.

## Код, пути и inline code

### Inline code

Использовать для:

- имен файлов;
- путей;
- названий функций;
- SQL-идентификаторов;
- переменных окружения;
- коротких терминов из кода.

Правила:

- шрифт: `Consolas` или равнозначный моноширинный;
- размер: 10-11 pt;
- цвет: темный, без синего оттенка;
- курсив: нет;
- жирность: только если нужно подчеркнуть именованную сущность, но не по умолчанию;
- не превращать inline code в декоративный элемент.

### Code Block

Для многострочного кода, SQL, конфигов и фрагментов логики:

- шрифт: `Consolas`;
- размер: 9.5-10 pt;
- цвет текста: темно-серый или черный;
- фон: очень светлый нейтральный, например теплый бежево-серый;
- левый и правый отступ: 0.6-0.8 см;
- межстрочный интервал: 1.0 или exact 12 pt;
- интервал до и после: 4-6 pt;
- переносы строк сохранять как есть;
- не использовать курсив;
- не использовать синий цвет для ключевых слов или путей.

Если код очень длинный, лучше уменьшить размер шрифта до 9 pt, чем ломать блок принудительными переносами.

## Цвета и акценты

Цветовая схема должна быть нейтральной и строгой.

### Базовая палитра

- Основной текст: `#111111`.
- Вторичный нейтральный: `#4A4A4A`.
- Линии, рамки, границы таблиц: `#C8C1B8`.
- Фон кодовых блоков и шапок таблиц: `#F6F3EC` или близкий светлый нейтральный оттенок.

### Акцентные цвета

Допустимы только два акцента:

- `Accent 1` для терминов и смысловых выделений: теплый терракотовый, например `#8A4B2A`.
- `Accent 2` для code-like сущностей, если нужен цветовой акцент: темный умброво-графитовый, например `#4B3F35`.

Правила использования акцентов:

- не окрашивать ими заголовки;
- не применять к целым абзацам без необходимости;
- не использовать для таблиц и декоративных элементов;
- не смешивать с голубым, бирюзовым, болотным и ярко-зеленым;
- не делать акцентом больше одной смысловой единицы в предложении, если это не список сущностей.

Если сомневаетесь, лучше оставить сущность черной и выделить ее только моноширинным шрифтом.

## Таблицы

Таблицы должны быть компактными, но не тесными.

### Базовые правила

- Заголовок таблицы располагать над таблицей.
- Шапку таблицы оформлять жирным шрифтом.
- Размер текста в таблице: 10-11 pt.
- Выравнивание текста в ячейках: по левому краю, числовые колонки можно выравнивать по правому краю.
- Вертикальное выравнивание: сверху.
- Отступы внутри ячеек: минимальные.
- Между абзацами в ячейке не оставлять лишнего воздуха.

### Оформление шапки

- Жирный шрифт.
- Светлая нейтральная заливка без синего оттенка.
- Тонкая рамка темно-серого или бежево-серого цвета.

### Длинные таблицы

- Разрешить повтор заголовка на новой странице.
- Не разрывать строку таблицы без необходимости.
- Если таблица становится слишком широкой, сначала сократить текст, а не увеличивать поля документа.

## Подписи

Для таблиц и рисунков использовать короткие подписи.

Правила:

- подпись должна быть лаконичной;
- цвет черный;
- размер 10 pt;
- без курсива по умолчанию;
- нумерация сквозная или по разделам, если это нужно проекту;
- подпись ставится перед таблицей или рисунком, если так проще воспринимается контекст.

## Приложение

Приложение допускается оформлять компактнее, чем основной корпус.

Рекомендуемые отличия:

- `Appendix Heading 1` вместо обычного `Heading 1`;
- основной текст 10 pt вместо 11 pt;
- межстрочный интервал 1.0-1.1;
- списки с чуть меньшими отступами;
- таблицы с более плотной компоновкой;
- кодовые блоки с уменьшенным вертикальным воздухом.

Приложение не должно выглядеть неряшливо, но может быть более плотным и служебным по тону.

## Практика для `python-docx`

Чтобы спецификация была удобно реализуема, рекомендуется придерживаться следующих технических правил:

- создавать и переиспользовать именованные стили документа;
- задавать параметры шрифта и абзацев через стиль, а не для каждого абзаца вручную;
- использовать `WD_ALIGN_PARAGRAPH.JUSTIFY` для основного текста;
- использовать `WD_BREAK.PAGE` или `page_break_before` для верхних разделов;
- задавать межстрочный интервал и интервалы абзацев через свойства paragraph format;
- не полагаться на ручные переносы строк для верстки;
- обновлять оглавление как поле, а не как статичный текст;
- для кодовых фрагментов применять моноширинный стиль по умолчанию;
- для акцентных цветов использовать ограниченный набор hex-значений;
- держать визуальные решения стабильными от документа к документу.

## Критерии приемки

Документ считается соответствующим спецификации, если:

- верхние разделы начинаются с новой страницы;
- оглавление рабочее и отражает реальные заголовки;
- заголовки черные, жирные и без курсива;
- основной текст визуально плотный, но читаемый;
- списки аккуратны и не перегружены отступами;
- код и пути моноширинные;
- голубой и болотный цвета не используются;
- акценты ограничены 1-2 спокойными оттенками;
- приложение выглядит компактнее, чем основной корпус;
- документ можно безболезненно обновлять и расширять через `python-docx`.
